📱 App Android — Defectos del Código (OkHttp / Kotlin)
7 defectos identificados en el reporte. Cada uno amplifica la carga en DB de forma distinta.
Defecto / Sección Estado actual ✓ Corrección Descripción del impacto
Timeout OkHttpClient
§2.2 — CRÍTICO
 30s connect=180s/read=180s/write=180s. Cada conexión lenta retiene el lease JDBC durante 3 minutos completos — el principal vector de amplificación.
runBlocking GPS
§2.1 — CRÍTICO (HeaderInterceptor)
 OFF runBlocking{} dentro del dispatcher OkHttp. Bloquea threads hasta 20–30s esperando GPS. En edificios, cada request puede quedar congelado.
Triple startLocationUpdates()
§2.3 — ALTA
 OFF (1 registro) Registrado en Application.onCreate() + MainActivity.onStart() + MainActivity.onResume(). Triplica callbacks GPS, aumenta freezes de red.
Endpoints duplicados
§2.4 — ALTA (ProxyApiService)
 OFF (unificado) verificarExistenciaCreditoActivo y obtenerUltimaVersionApp declarados 2× con distintas signatures. ViewModels pueden invocar ambas versiones → +25% req/s.
Carga paralela al login
§10.3 — MEDIA
 OFF (diferida) Al login: 6–10 requests en paralelo (perfil + créditos + notificaciones + versión + …). Spike ×2.2 en los primeros 8s de sesión.
401 sin cancelar en vuelo
§UnauthorizedInterceptor — ALTA
 OFF (cancel all) Cuando un 401 limpia tokens, las llamadas en vuelo NO se cancelan. Siguen consumiendo su lease JDBC hasta que agotan su timeout completo (180s).
Session polling (onResume)
§H7 — MEDIA (sessionViewModel)
 OFF (lazy) sessionViewModel.checkSession() en cada onResume de Activity. Genera tasa de fondo constante independiente de acciones del usuario (+0.3 req/s por usuario).
Retries en ViewModels
§H6 — MEDIA (comportamiento inferido)
 OFF (no retry) Al recibir error por ConnectionReserveTimeout=10s, ViewModels pueden reintentar. 40% de requests rechazados vuelven → amplificación en cascada.
💡 El timeout de 180s es el principal amplificador. Cada request lento retiene el lease JDBC durante 3 minutos.
